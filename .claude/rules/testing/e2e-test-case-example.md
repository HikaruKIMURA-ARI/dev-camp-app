## いい E2E テストケースの具体例

以下のように、`test.describe` を使い抽象度の順にインデントされている、生きたドキュメントにすること。
3 年後アサインされた人が見ても、何を目的にテストしているかがわかる抽象度の高い `test.describe` を書き、その具体のユーザー操作と画面の変化を `test` として記述すること。

E2E はテストピラミッドで最少（ハッピーパス 2-3 + 異常系 1-2）。単体・結合でカバー済みの検証は重複させない。

### Phase 1: シナリオ列挙（`test.skip()` のみ）

```typescript
import { test } from "@playwright/test";
import { truncateAll } from "./fixtures/db";

// 機能要件ごとに spec ファイルを分ける（例: tests/e2e/event.spec.ts）
test.describe("イベント日程調整", () => {
  // 仕様を抽象度の高い状態のユーザーストーリーで分類する
  // 3年後アサインされた人が、テストをドキュメントとして参照でき、配下のシナリオが何を検証するかがわかるようにする
  test.describe("主催者がイベントを作成したとき詳細ページに遷移する", () => {
    // 抽象度の低い状態の具体的なシナリオを入れる（Phase 1 は test.skip のみ。コードは書かない）
    test.skip(
      "タイトル・候補日時・回答締め切りを入力して送信すると、詳細ページへ画面遷移し入力内容が表示されること"
    );
    // 候補日時の表示形式は結合テストで検証済みなので、締め切りなしの別パターンは入れない
  });

  // 仕様を抽象度の高い状態のユーザーストーリーで分類する
  test.describe("参加者が回答を送信したとき集計表が更新される", () => {
    test.skip(
      "名前と候補への回答を送信すると、htmx で #responses が差し替わり参加者名と集計が表示されフォームがリセットされること"
    );
    test.skip(
      "集計表の編集ボタンを押して名前を変更し更新すると、差し替え後の集計表に新しい名前が反映されること"
    );
    // POST の 201 や DB 保存は結合テストで検証済み。ここではブラウザ上の差し替えと表示に絞る
  });

  // 仕様を抽象度の高い状態のユーザーストーリーで分類する
  test.describe("不正な操作や存在しないリソースにアクセスしたときエラーが画面に示される", () => {
    test.skip(
      "存在しないイベント ID にアクセスすると 404 ページが表示されること"
    );
    test.skip(
      "タイトル未入力のまま作成ボタンを押すと、HTML5 required によりフォームが送信されず作成画面に留まること"
    );
    // 422 のエラーメッセージ文言は単体テストで検証済み。ここでは画面遷移が起きないことだけを観測する
  });
});
```
