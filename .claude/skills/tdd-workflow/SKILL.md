# TDD ワークフロー

あなたはテスト駆動開発（TDD）の厳格な実践者です。
ルートの `CLAUDE.md` と `.claude/rulese/testing/test-philosophy.md` を読んだ上で、のテスト思想に必ず従い、
すべてのコード変更は以下のサイクルに従ってください。

## Phase 1: テストケース列挙

1. `.claude/agents/test-case-writer`のサブエージェントを使うこと
2. 要件を分析し、テストケースを設計する
3. 生きたドキュメントになるようにテストケースだけを列挙する(テストケースとテストコードを同時に書かない)
4. 列挙が完了したら必ずユーザーのレビューゲートを挟むこと。勝手に次のフェーズに進まないこと

## Phase 2: RED（テストコード作成）

1. `.claude/agents/test-code-writer`のサブエージェントを使うこと
2. 要件を分析し、テストケースを設計する
3. 失敗するテストを 1 つ書く
4. テストを実行して失敗を確認する（`bun test`）
5. 失敗メッセージが期待通りであることを確認する
6. 完了したら必ずユーザーのレビューゲートを挟むこと。勝手に次のフェーズに進まないこと

## Phase 3: GREEN（最小実装）

1. `.claude/agents/tdd-implementer`のサブエージェントを使うこと
2. テストをパスさせる最小限のコードを書く
3. "最小限"とは：余分な機能、最適化、リファクタリングを含まない
4. テストを実行してパスを確認する
5. 他のテストが壊れていないことを確認する

## Phase 4: REFACTOR（改善）

1. `.claude/agents/tdd-refactorer`のサブエージェントを使うこと
2. `.claude/agents/ui-optimizer`のサブエージェントを使い UI のリファクタリングを行うこと
3. コードの重複を除去する
4. 命名を改善する
5. 関数を適切なサイズに分割する
6. テストを実行して全てパスすることを確認する
7. 完了したら必ずユーザーのレビューゲートを挟むこと。勝手に次のフェーズに進まないこと

## 制約

- レビュー時はどのテストケースが対象かを明示すること
- Phase 1 を完了するまで Phase 2 に進まない
- Phase 2 を完了するまで Phase 3 に進まない
- Phase 3 を完了するまで Phase 4 に進まない
- 各 Phase でテスト実行を必ず行う
- テストが失敗したまま次のテストを追加しない
- 各フェーズでレビューゲートが支持されている場合、それに従い勝手に次に進まないこと
