# Requirements Document

## Project Description (Input)

tyousei-ph2 — 既存の出欠調整機能（イベント作成 / 閲覧 / 回答送信 / 編集 / テーマ切替）の上に乗せる「参加者カード（遊戯王風）」拡張。回答送信時に参加者名から AI でテキストカードを生成し、イベント詳細ページ上部のカルーセルに追加する。本リポジトリには現時点で Slack 催促や AI による文章生成機能は実装されておらず、AI API クライアントの導入も本仕様が初回となる。

## Introduction

本仕様は、既に動作している出欠調整機能（イベント作成 / 閲覧 / 回答送信 / 編集 / テーマ切替）の上に乗せる Phase 2 機能として、**参加者カード**を追加する。参加者が回答を送信するタイミングで、その参加者名から **遊戯王風のテキストカード**（イラスト枠なし）を AI で自動生成し、イベント詳細ページ上部に横並びで追加する。カードは「二つ名 / レアリティ / 属性 / 種族 / フレーバーテキスト / 攻撃力 / 守備力」の 7 属性を持ち、回答テーブルと並列に表示することで、参加者の登場感と遊び心を演出する。

AI API クライアントの導入は本仕様が初回であり、既存に Gemini 等の AI 統合パターンは存在しない。本仕様で 3 段フォールバック戦略（Tier 1: AI 生成 / Tier 2: テンプレ生成 / Tier 3: 最低限デフォルト）と無料枠枯渇検知のプロセス内フラグ管理を新規に導入する。

本機能はあくまで装飾的拡張であり、出欠調整の主機能（候補選択・集計）の挙動は変えない。AI が利用できない状況（無料枠枯渇・タイムアウト・ネットワーク不全）でも回答送信は常に成立し、カードは決定論的なフォールバックで生成される。

## Boundary Context

- **In scope**:
  - 参加者カードの 7 属性（二つ名 / レアリティ / 属性 / 種族 / フレーバーテキスト / 攻撃力 / 守備力）の自動生成
  - 既存 `event_responses` 行に 1:1 で紐づくカードデータの永続化
  - イベント詳細ページ上部の横並びカルーセル UI（回答テーブルとは別領域）
  - レアリティに応じたカード枠の視覚差分（色 / 装飾）と属性・種族の表示
  - Gemini API（無料枠の軽量モデル）による属性生成と、3 段フォールバック戦略
  - プロンプトインジェクション対策（参加者名の引用区切り）
  - htmx フラグメント差し替えとカルーセル更新の連動
  - 初回回答送信時のみカード生成（編集ではカードを再生成しない）
- **Out of scope**:
  - カードのイラスト・画像生成（テキスト + 装飾のみ）
  - カード単独の永続 URL / SNS 共有 OGP
  - カード同士のバトル / コレクション / 並べ替え / ゲーム性
  - カードのユーザー編集（再生成ボタン含む）
  - 主催者によるカード削除 / 非表示制御
  - カード生成履歴の保存（再生成は行わないため）
  - AI モデルの選択 UI / プロンプトのユーザー編集
  - Gemini API の有料枠 / 課金前提運用
  - カードの多言語対応（日本語のみ）
- **Adjacent expectations**:
  - 本仕様で初めて AI API クライアントをプロダクションコードに導入する。既存に同様の統合がない前提でモジュール境界・設定・テスト戦略を設計する
  - 既存の「初回 GET はフルページ / htmx 経由はフラグメント」規約を守る
  - 既存テーブル（`events` / `event_options` / `event_responses` / `event_option_responses`）には破壊的変更を加えない
  - `schema.ts` に骨格として `slackWebhooks` テーブル定義が存在するが、本仕様はこのテーブルを利用しない（参照・拡張ともに行わない）

## Requirements

### Requirement 1: 参加者カードの自動生成

**Objective:** As a イベント参加者, I want 自分が回答を送信した瞬間に名前から遊戯王風カードが生成される, so that 出席表明が楽しい体験になる

#### Acceptance Criteria

1. When 参加者が `POST /events/:id/responses` で新規回答を送信したとき、the Card Service shall 当該回答に対応するカードを 1 件生成し、回答と同一トランザクションで永続化する
2. The Card Service shall カードに次の 7 属性を保持する: 二つ名（参加者名を含む短い称号文字列）/ レアリティ（短い文字列。推奨候補は UR / SR / R / N だが、AI が稀にこれ以外のレア表記を返した場合もそのまま受け入れてよい）/ 属性（短い文字列。推奨候補は 光 / 闇 / 水 / 風 / 地 / 火 だが、AI が稀にそれ以外を返してもよい）/ 種族（短い文字列。推奨候補は 戦士 / ドラゴン / 魔法使い / アンデッド / 悪魔 / 幻獣 / 魚 / サイバー だが、AI が稀にそれ以外を返してもよい）/ フレーバーテキスト（1 行の自由記述）/ 攻撃力（0 以上の整数）/ 守備力（0 以上の整数）
3. The Card Service shall カードを当該回答（`event_responses`）に 1:1 で紐づけ、回答 1 件につきカード 1 件を保証する
4. When 参加者が既存の回答を `PUT /events/:id/responses/:responseId` で更新したとき、the Card Service shall 当該回答に紐づくカードを再生成せず、既存カードをそのまま保持する
5. If 参加者の表示名と無関係に二つ名が生成されてしまうとき、then the Card Service shall 二つ名の文字列に参加者名そのものを必ず含めて永続化する
6. The Card Service shall 二つ名・フレーバーテキスト・属性値が極端に長くなる場合に備えて、各文字列を所定の上限文字数で切り詰めた上で永続化する

### Requirement 2: 参加者カード一覧のカルーセル表示

**Objective:** As a イベントページの閲覧者, I want 参加者カードを画面上部に横並びで一覧できる, so that 誰が回答済みかと各人の個性を一目で把握できる

#### Acceptance Criteria

1. When 利用者が `GET /events/:id` にアクセスしたとき、the Event Page shall 既存の候補一覧 / 回答テーブルより上部に、参加者カードを横並びで並べたカルーセル領域を表示する
2. While 回答件数が 1 件以上であるとき、the Event Page shall 各参加者のカード（二つ名 / レアリティ / 属性 / 種族 / フレーバーテキスト / 攻撃力 / 守備力）をカルーセル内に 1 枚ずつ描画する
3. While 回答件数が 0 件であるとき、the Event Page shall カルーセル領域を表示しないか、または「まだ参加者がいません」相当の空状態メッセージを表示する
4. When 参加者が新規回答を送信したとき、the Event Page shall htmx の差し替えと連動してカルーセルにも新しいカードを追加する
5. The Event Page shall カルーセル内のカードを回答送信順（古い参加者から新しい参加者の順）で並べる
6. When カルーセルに収まりきらない数のカードが存在するとき、the Event Page shall 横スクロール、横スワイプ、または横送りボタンのいずれかでスクロール可能にする

### Requirement 3: 遊戯王風カードのビジュアル仕様

**Objective:** As a イベントページの閲覧者, I want カードがレアリティ・属性で見分けられる装飾を持つ, so that 一覧の中でレア参加者を直感的に発見できる

#### Acceptance Criteria

1. The Card View shall 各カードに二つ名 / レアリティ / 属性 / 種族 / フレーバーテキスト / 攻撃力 / 守備力をすべて視覚的に提示する
2. The Card View shall レアリティ（UR / SR / R / N）に応じてカード枠の色・装飾を変える
3. The Card View shall 属性（光 / 闇 / 水 / 風 / 地 / 火）を、文字またはピクト的なバッジで識別可能に表示する
4. The Card View shall 種族を文字列として表示する
5. The Card View shall 攻撃力と守備力を `ATK / DEF` 風の数値ペアとして表示する
6. The Card View shall イラスト枠を必須としない（テキスト主体のレイアウトで成立する）
7. The Card View shall 既存 pico.css の配色変数を尊重し、ライト / ダーク両テーマで視認可能な配色を採用する
8. The Card View shall スクリーンリーダー利用者向けに、カード 1 枚の意味的なラベル（少なくとも二つ名 + 参加者名）を取得できるようにする

### Requirement 4: AI 生成の信頼性とフォールバック

**Objective:** As a イベント主催者, I want AI が使えない状況でも回答送信が常に成立しカードが生成される, so that 主機能（出欠調整）が外部 API に巻き込まれて壊れない

#### Acceptance Criteria

1. When 回答送信時に AI 生成リクエストが所定のタイムアウト時間内に成功したとき、the Card Service shall そのレスポンスから 7 属性を抽出し Tier 1（AI 生成カード）として永続化する
2. If AI 生成リクエストがタイムアウト・ネットワークエラー・5xx・JSON 不正・スキーマ不一致のいずれかで一時的に失敗したとき、then the Card Service shall 二つ名テンプレート（例: 「疾風の出席者 ${名}」「酒豪王 ${名}」等）と固定候補からランダム抽選した属性で Tier 2（テンプレフォールバック）カードを永続化する
3. If AI 生成リクエストが 429（レート制限）または 403（クォータ超過）等で無料枠枯渇相当のエラーを返したとき、then the Card Service shall 二つ名を「${名}」のみ、レアリティ / 属性 / 種族 / フレーバーテキスト / 攻撃力 / 守備力をすべて最低限のデフォルト値とする Tier 3（最低限デフォルト）カードを永続化し、当該プロセス内で以降の AI 呼び出しを抑止する
4. The Card Service shall いずれの Tier であってもカードを必ず 1 件永続化し、回答送信フローを失敗扱いにしない
5. The Card Service shall AI へ送るプロンプトにおいて、参加者名を引用区切り（または同等の構造的境界）で囲み、参加者名がプロンプトの指示部分として解釈されないようにする
6. The Card Service shall AI から受け取った応答に対し、レアリティ / 属性 / 種族の各項目が存在することと文字列型であることを検証する（推奨候補リストに含まれない値が返っても、サニタイズ済みであればそのまま受け入れる。空文字・改行・制御文字・極端な長さのみフォールバックに差し替える）
7. The Card Service shall 攻撃力・守備力が負数または異常に大きな値（所定の上限超過）であった場合に、所定の範囲にクランプして永続化する
8. While プロセス内クォータ枯渇フラグが立っているとき、the Card Service shall 以降の回答送信で AI を呼び出さず Tier 3 で永続化を続ける

### Requirement 5: カードデータの永続化と取得

**Objective:** As a 開発者, I want カードデータが既存スキーマと素直に同居する, so that マイグレーションの破壊性を抑えつつ既存テストとの互換を保てる

#### Acceptance Criteria

1. The Card Service shall カードの 7 属性を `event_responses` 行と 1:1 で紐づく形で永続化する（同テーブル列追加または別テーブル + FK のいずれか）
2. The Card Service shall 既存テーブル（`events` / `event_options` / `event_responses` / `event_option_responses`）の既存列・既存インデックス・既存制約を破壊的に変更しない
3. When `getEventWithOptions(eventId)` 相当の取得処理が呼ばれたとき、the Card Service shall 当該イベントの全回答に対するカードを併せて返す
4. The Card Service shall カードと回答の永続化を同一トランザクションで行い、片方だけが残る不整合を生まない
5. If 当該回答にカードが紐づいていない状態が万一発生したとき、then the Event Page shall その参加者をカルーセルから除外せず、カード位置にフォールバック表示（例: 「カードを生成中…」または既定カード）を行う

### Requirement 6: htmx 連動とフラグメント規約の遵守

**Objective:** As a 利用者, I want 回答送信後にページ全体を再読み込みせずカードと回答テーブルの両方が更新される, so that 既存の即時反映 UX が損なわれない

#### Acceptance Criteria

1. When 参加者が回答を送信したとき、the Event Page shall 回答テーブル領域（既存 `#responses`）とカルーセル領域の両方を htmx で部分更新する
2. The Event Page shall 回答送信レスポンスとして、回答テーブルとカルーセルの両方を含む 1 つのフラグメント、または `hx-swap-oob` 等で 2 領域を同時更新できる形のフラグメントを返す
3. The Event Page shall フルページの初回 GET 時にもカルーセルが正しく描画される（フラグメント専用にしない）
4. While 回答送信中であるとき、the Event Page shall ボタンの多重押下によってカードと回答が重複生成されないよう、サーバ側で同一参加者の同一イベントへの即時連続送信を 1 件として扱うか、クライアント側で多重送信を抑止する
5. The Event Page shall カルーセル領域に固有の ID（例: `#cards`）を持たせ、htmx の `hx-target` で一意に指定できるようにする

### Requirement 7: AI プロンプトと出力スキーマの安定性

**Objective:** As a 開発者, I want AI 出力をプログラマブルに扱える形に固定する, so that フォールバック判定とテストが書きやすい

#### Acceptance Criteria

1. The Card Service shall AI へ「二つ名 / レアリティ / 属性 / 種族 / フレーバーテキスト / 攻撃力 / 守備力 を含む JSON」を返すよう指示するプロンプトを構築する
2. The Card Service shall AI 応答を JSON としてパースし、7 属性すべてが揃わない場合は不正応答として扱う
3. The Card Service shall レアリティ・属性・種族について、推奨候補リスト（レアリティ: UR / SR / R / N、属性: 光 / 闇 / 水 / 風 / 地 / 火、種族: 戦士 / ドラゴン / 魔法使い / アンデッド / 悪魔 / 幻獣 / 魚 / サイバー など）をプロンプト内で提示しつつ、「たまに推奨外の値を返してもよい」旨を明示し、DB スキーマは自由文字列で保持する（厳格な enum 制約をかけない）
4. The Card Service shall フレーバーテキストに改行・制御文字が混入していた場合は、表示前にスペースに置換するか除去して 1 行表現を維持する
5. The Card Service shall AI に渡すモデル名・温度・最大トークン数等を環境変数または設定で差し替え可能にし、テスト時には実 API を呼ばずに済むよう抽象化する
